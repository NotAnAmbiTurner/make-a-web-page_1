An HTML Document is Just a Bunch of Text

Since an HTML document essentially consists of just a bunch of text, it is at the mercy of the interpreter.

Accordingly, some funky simplifying rules have been developed, which help create a semi-reliable way of rendering HTML into graphics accross platforms - it's not perfect, but still a really amazing level of coordination if you compare it to pretty much anything other human endeavour. Some of these rules are:

  • Everything is a box
  • Elements nest within one-another, with the most local rules supercedeing the more global
  • Each element may have one or more specified styles

The Importance of Avoiding Repetition

"I'm so DRY right now." I had a friend that used to say that all the time... I had no idea that she was a programmer. *rimshot*
... but seriously, that's a thing, from the east coast in Canada, and it totally doesn't mean what you think... or maybe it does.

I don't like the complexity, or the redundancy that comes with repeating myself -- how I am I supposed to create really cool things if I have to be redundant?

I would much rather think in rules, boxes, abstractions, than spend my time re-doing something that a robot could have done.

Don't repeat yourself isn't a programming principle, it's a way of life. In a structured program (i.e. one which works in a heirarchical manner using flow control, appropriate abstraction, and blocks, as opposed to using various goto statements, etc.) which avoids repetition, adn is much easier to read. In turn, it becomes more practical to:

  • maintain and alter the code -- if you have to change the way something functions, either because it breaks or it needs to work a different way, and it's abstracted, you only have to change the one function instead of changing each separate execution within the code.
  • maintain uniformity -- for example, to change the heading colors in this document, I would only have to update the h2 section in my CSS file, as opposed to changing every STYLE parameter in every heading. This also makes it easier to add new headings and maintain the visual uniformity of the page.

Basic HTML and CSS Knowledge

Things I know and find useful include:

  • Inheritance:
    • More specific (lower-level) items supercede higher-order items, just like classes and subclasses. This means in-line > block styles. For the record, I don't think it's a coincidence that CSS uses curly-braces.
    • Everything is easier the more centralized it is -- defaulting to higher-order styling looks better because it's more consistent and is easier to manage. If we consider that things that look familiar tend to look better, does that create a self-fulfilling prophecy? I say yes.
    • You can also earn money using this kind of inheritance.
    • It is so interesting that comments within the STYLE tags require CSS-style commenting
  • Tools:
    • I started this particular project on Codepen. That was a pain, SublimeText is just way easier to work with.
    • The Mozilla reference is ridiculous and hard to use, I much prefer to Google "#{query} w3school".
